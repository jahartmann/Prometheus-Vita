# Deployment

Reference deployment for a small Proxmox cluster (1-10 nodes, single-host
Prometheus instance). Everything else — HA Postgres, multi-replica Prometheus,
Kubernetes — is outside scope.

## Prerequisites

- Linux host with Docker 24+ and the Compose v2 plugin
- Outbound network access to your Proxmox API endpoints
- For TLS with a real cert: a DNS A/AAAA record pointing at the host

## First-time setup

```sh
git clone <repo>
cd Prometheus-Vita
./setup.sh                 # generates strong secrets in .env
docker compose up -d       # core stack (postgres, redis, backend, frontend)
```

Open `http://<host>:3000`. The admin password printed by `setup.sh` is the
initial login. Change it immediately under *Einstellungen → Benutzer*.

## Production checklist

Run through each item before exposing the instance:

- [ ] `.env` exists and **none** of `JWT_SECRET`, `ENCRYPTION_KEY`,
  `POSTGRES_PASSWORD`, `REDIS_PASSWORD` are still `changeme_*` placeholders
  (the backend will refuse to start otherwise).
- [ ] `LOG_LEVEL=info` and `LOG_FORMAT=json` in `.env` so logs are aggregable.
- [ ] TLS profile enabled — see [TLS / reverse proxy](#tls--reverse-proxy).
- [ ] Backup profile enabled and a recent dump exists — see
  [BACKUP.md](BACKUP.md).
- [ ] `docker compose ps` shows every container as `(healthy)`.
- [ ] `curl -fsS http://<host>:8080/ready` returns 200.
- [ ] Admin account password rotated and unused seed account disabled.
- [ ] `PROXMOX_TLS_INSECURE=false` once a Proxmox CA cert is configured
  (`PROXMOX_TLS_CA_CERT=/path/to/ca.pem`).

## TLS / reverse proxy

Caddy ships as an opt-in profile so local dev works without it.

### Local / LAN deploy (self-signed cert)

```sh
# .env additions
PROMETHEUS_HOSTNAME=prometheus.lan
PROMETHEUS_TLS_SNIPPET=tls_internal
```

```sh
docker compose --profile tls up -d
```

Browser will warn on first visit (internal CA); accept once. The cert is
generated by Caddy and stored in the `caddy_data` volume — it persists across
restarts.

### Public deploy (Let's Encrypt)

Requirements:

- A real DNS record (`prometheus.example.com`) pointing at the host
- Ports 80 and 443 open from the public internet to the host
- A contact email for ACME registration

```sh
# .env additions
PROMETHEUS_HOSTNAME=prometheus.example.com
PROMETHEUS_TLS_SNIPPET=tls_public
CADDY_ACME_EMAIL=admin@example.com
```

```sh
docker compose --profile tls up -d
```

Caddy will provision and renew a Let's Encrypt cert automatically. First-time
issuance can take up to a minute; watch with `docker compose logs -f caddy`.

When TLS is enabled, **direct access to ports 3000 and 8080 should be blocked
at the firewall**. The compose file does not bind them to `127.0.0.1` because
that's a per-deployment decision; do it via your host firewall (ufw,
nftables, cloud SG, …) or remove the `ports:` blocks of `backend` and
`frontend` if Caddy is the only ingress.

## Updating

```sh
git pull
docker compose pull         # for any pinned upstream images
docker compose up -d --build
```

Database migrations run automatically on backend startup. The schema-lock
table plus a Postgres advisory lock prevent concurrent migration runs when
multiple replicas boot at once. Migrations are forward-only — there is no
automated down-migration. To roll back, restore the matching backup (see
BACKUP.md) and re-deploy the previous container image.

## Observability

- Backend logs: JSON on stdout, level controlled by `LOG_LEVEL`.
- Health probes:
  - `/live` — process is alive (cheap, returns 200 unconditionally)
  - `/ready` — Postgres + Redis are reachable (returns 503 when not)
  - `/health` — legacy combined endpoint
- WebSocket hub: drop counter is exposed in code via `wsHub.DroppedCount()`;
  surface it through a metrics endpoint if you need an alert on it.

## Resource sizing

The defaults in `docker-compose.yml` target a small/medium cluster:

| Service  | CPU limit | Memory limit | Notes |
| -------- | --------- | ------------ | ----- |
| postgres | 2 | 1 GiB | bump memory before CPU when adding nodes |
| redis    | 1 | 384 MiB | with `maxmemory 256mb` + LRU eviction |
| backend  | 2 | 1 GiB | scales with number of nodes scraped |
| frontend | 1 | 512 MiB | static after first compile |

For 30+ nodes, expect to double the postgres and backend allowances.

## Reverting a bad release

1. `docker compose down` — stops everything but keeps volumes.
2. Restore the most recent dump per [BACKUP.md](BACKUP.md).
3. Check out the previous git tag and `docker compose up -d --build`.
